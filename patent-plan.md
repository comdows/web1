## 1. BM(영업방법) 특허가 성립하는 조건

순수한 영업 아이디어("익명으로 매물을 게시하고 소개한다")는 특허 대상이 아닙니다.
**소프트웨어에 의한 정보처리가 하드웨어(컴퓨터) 상에서 구체적으로 실현**되어야 발명으로 인정됩니다
(특허청 컴퓨터 관련 발명 심사기준). 즉 청구항을 다음 형태로 잡아야 합니다:

- ~를 수행하는 **방법** (서버/단말이 수행하는 단계들)
- ~를 포함하는 **시스템/장치**
- ~를 실행시키는 **컴퓨터 판독 가능 기록매체에 저장된 프로그램**

그래서 아래 발명 3건은 전부 "무엇을 어떤 데이터로 어떻게 판정·전이·통지하는가"라는
**구체적 정보처리 구성**으로 기술했습니다.

## 3. 발명 1 — 명세서 초안

### 발명의 명칭
**플랫폼 디렉토리 데이터베이스를 이용한 익명 거래 게시물의 식별정보 누출 자동 검증 방법 및 시스템**

### 기술분야
온라인 익명 거래 게시판(사업·자산 양수도 리스팅 등)에서 게시물 텍스트에 포함된
작성자 식별 가능 정보를 자동으로 탐지·차단·경고하는 정보처리 기술.

### 배경기술
익명 매물 게시판은 작성자가 무심코 적은 플랫폼명·도메인·정확한 수치로 신원이 유추되어
익명성이 붕괴되는 문제가 있다. 종래에는 운영자의 수기 검수에 전적으로 의존하여
① 검수 지연 ② 검수자 숙련도 편차 ③ 작성 시점의 교정 기회 부재라는 한계가 있었다.
연락처 패턴(전화번호·이메일)의 정규식 차단은 알려져 있으나, **작성자가 속한 도메인의
고유명사 사전과 대조해 신원 유추 위험을 판정**하는 구성은 제공되지 않았다.

### 해결하려는 과제
게시물 접수 시점과 검수 시점 양쪽에서, 식별정보 누출 위험을 위험 등급별로 자동 판정하여
익명성 붕괴를 예방하고 검수 부담을 줄이는 것.

### 과제의 해결 수단 (핵심 구성)
1. **검증 사전의 구성**: 서비스가 보유한 **플랫폼 디렉토리 데이터베이스**(등재 플랫폼명 N건)에서
   일정 길이 이상의 등재명을 추출해 식별정보 검증 사전을 자동 구성한다.
   (사전을 별도 구축하지 않고 서비스의 기존 데이터 자산을 재사용 — 등재가 늘수록 검증력이 함께 증가)
2. **2계층 판정**: 게시물 텍스트를
   - **차단 계층**(연락처·메신저 ID·URL 패턴 — 서버 측 제약조건과 동일 기준의 이중화)과
   - **경고 계층**(ⓐ 도메인 표기 ⓑ 단위 결합 수치[억·만 원·만 명·%] ⓒ 검증 사전 일치)으로 판정한다.
3. **2단 제출 확인**: 차단 계층 검출 시 접수를 거부하고, 경고 계층만 검출되면 위험 근거
   (검출 문자열·사유)를 표시한 뒤 **작성자의 재확인 입력이 있어야** 접수를 완료한다.
   텍스트가 수정되면 재확인 상태를 초기화하여 항상 최신 텍스트 기준으로 판정한다.
4. **검수자 하이라이트**: 관리 화면에서 접수 원문과 게시 초안 양쪽에 대해 동일 판정을 재수행하여
   위험 근거를 하이라이트한다(게시 직전 최종 방어선).
5. **서버 측 제약과의 이중화**: 차단 계층 패턴은 데이터베이스 체크 제약조건과 동일하게 유지되어
   클라이언트 우회 시에도 저장이 거부된다.

### 발명의 효과
- 익명성 붕괴 사고를 접수 시점에 예방(사후 삭제 대비 피해 최소화)
- 디렉토리 성장 = 검증 사전 성장의 선순환
- 검수자 1인 운영에서도 일관된 검수 품질
- 작성자에게 즉시 교정 기회 제공 → 반려율 감소

### 도면의 간단한 설명 (출원 시 흐름도로 작성)
- 도1: 시스템 구성(단말–검증 모듈–디렉토리 DB–검수 화면)
- 도2: 2계층 판정·2단 제출 흐름도
- 도3: 검수 화면 하이라이트 예시

### 실시예
본 리포지토리 `app/src/lib/anonymity.ts`(판정 모듈), `app/src/pages.tsx` SellForm(2단 제출),
`app/src/admin.tsx` DealSubQueue(검수 하이라이트), `backend/migrations/0005_intro.sql`
chk_*_nocontact(서버 이중화)가 구현 실시예에 해당.

### 청구범위 초안 (변리사 보정 전제)
1. **[독립항·방법]** 서버 또는 단말이, (a) 플랫폼 디렉토리 데이터베이스로부터 등재명을 추출하여
   검증 사전을 구성하는 단계; (b) 사용자 게시물 텍스트를 연락처 패턴의 차단 계층과, 도메인 표기·
   단위 결합 수치·상기 검증 사전 일치를 포함하는 경고 계층으로 판정하는 단계; (c) 차단 계층 검출 시
   접수를 거부하고, 경고 계층 검출 시 검출 근거를 표시한 후 사용자의 재확인 입력을 조건으로
   접수하는 단계;를 포함하는 익명 게시물의 식별정보 누출 검증 방법.
2. 제1항에 있어서, 텍스트 수정 시 상기 재확인 상태를 초기화하는 단계를 더 포함하는 방법.
3. 제1항에 있어서, 관리자 검수 화면에서 접수 원문과 게시 초안 각각에 대해 상기 판정을 재수행하여
   검출 근거를 표시하는 단계를 더 포함하는 방법.
4. 제1항에 있어서, 상기 차단 계층의 패턴이 데이터베이스 체크 제약조건과 동일하게 유지되어
   클라이언트 우회 시 저장이 거부되는 방법.
5. **[독립항·시스템]** 상기 방법을 수행하는 검증 모듈·디렉토리 데이터베이스·검수 인터페이스를
   포함하는 익명 거래 게시 시스템.
6. **[독립항·기록매체]** 제1항 내지 제4항 중 어느 한 항의 방법을 컴퓨터에서 실행시키기 위한
   프로그램을 기록한 컴퓨터 판독 가능 기록매체.

## 4. 발명 2 — 명세서 초안 (요지)

### 발명의 명칭
**쌍방 동의 이력 기반 단계적 연락처 공개 방법 및 시스템**

### 핵심 구성
1. 매도자·매수자 각각의 **연락처 공유 동의 시각을 데이터 필드로 기록**(매도자: 접수 시,
   매수자: 관심 등록 시 — 동의 없는 레코드는 소개 불가로 판정).
2. 소개 절차를 상태머신으로 관리: `관심 접수 → ①매도자 의사 확인(상대 정보는 익명 요지만 전달)
   → ②소개 발송(양측 이메일 상호 공개 + 후속 절차 안내 포함) → 소개 완료(시각·수행자 기록)`.
3. **게이트 조건**: ②소개 발송 행위가 선행되지 않으면 '소개 완료' 상태 전이가 불가;
   양측 이메일과 동의 이력이 모두 존재해야 ②가 활성화.
4. **구조적 비노출**: 이메일은 관리자 전용 뷰(행 수준 접근 제어)에서만 조회되고
   공개 조회 경로에는 작성자 식별자 자체가 포함되지 않는다.

### 효과
동의 없는 개인정보 제공 원천 차단(개인정보보호법 정합), 소개 이행 시점의 증적 확보
(향후 과금·환불 판정 근거), 일방 통보식 소개로 인한 분쟁 예방.

### 청구항 방향
방법 독립항(동의 시각 기록 → 게이트 판정 → 단계 전이) + 상태머신 종속항 + 뷰 분리 종속항 + 시스템·기록매체 항.

### 실시예
`0004_open.sql`/`0005_intro.sql`(contact_consent_at, introduced_at/by, v_admin_intro_queue),
`admin.tsx` IntroQueue(2단 mailto + 게이트), `pages.tsx` 동의 체크(접수·관심 폼).

## 5. 발명 3 — 명세서 초안 (요지)

### 발명의 명칭
**익명 리스팅과 조건 프로파일의 비식별 자동 대조 및 통지 방법**

### 핵심 구성
1. 매수 희망자가 관심 분야·예산 밴드·거래 형태로 구성된 **조건 프로파일(브리프)**을 등록.
2. 익명 리스팅은 코드명·범주·밴드만 담은 **비식별 공개 뷰**로 제공.
3. 대조 연산을 서버가 아닌 **이용자 단말(클라이언트)에서 수행** — 서버는 누가 무엇에
   매칭되는지 알 필요가 없어 개인정보 처리를 최소화(분야 일치 ∧ 형태 호환 판정).
4. 매칭 결과는 이용자 개인 화면(내 활동)에 배너로 통지하고, 관리자 화면에는 브리프별
   매칭 코드명을 표시하여 수동 안내를 보조.

### 효과
알림 인프라 없이 "조건 매칭 안내" 이행, 개인정보 최소 처리(privacy by design), 매칭 로직의
서버 부하 제로.

### 청구항 방향
방법 독립항(비식별 뷰 + 클라이언트 대조 + 개인 화면 통지) + 형태 호환 판정 종속항 + 시스템·기록매체 항.

### 실시예
`lib/api.ts` briefMatchesDeal·v_deals_public, `account.tsx` 매칭 배너, `admin.tsx` IntroQueue 브리프 매칭 표시.

## 6. 출원 실행 체크리스트
