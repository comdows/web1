## 1. Executive Summary

**문제.** 사업자·개인이 "나가서 붙을 플랫폼"(판매채널·입점처·자금조달처·매칭처)을 고를 때,
정보가 블로그·유튜브·카페에 파편화돼 탐색 비용이 크다. Capterra·G2는 *내부에서 쓰는
소프트웨어*만 다루고, *외부에 붙는 플랫폼*은 아무도 구조적으로 분류·비교하지 않는다.

**해법(3단계).**
- **1단계(Wedge) 발견·비교·평가:** 플랫폼을 구조적으로 분류하고 수수료·정산주기·입점조건·정산리스크를 표준화 비교. 수요측 트래픽을 SEO로 확보.
- **2단계 제휴 매칭(BD as a Service):** 플랫폼 간 가입자 교환·송출·정산·협업 중개.
- **3단계 M&A·엑시트 거래소:** 동종 플랫폼 매매·엑시트 수요 연결.

**타깃(1단계).** 1차: 플랫폼을 고르려는 사업자/개인(수요). 2차(데이터 소스): 플랫폼 운영자.

**차별점/해자.** ① 아무도 표준화하지 않은 "외부 부착형 플랫폼" 비교 스키마, ② 이음의 다국어·프로그래매틱 SEO 인프라로 만든 롱테일 방어막, ③ 1단계 트래픽이 2·3단계 양면 유동성 씨앗이 되는 구조적 연결.

**North Star.** *주간 "의사결정 도달 비교 세션 수"*(≥2개 플랫폼 비교 + 상세/외부이동/계산 실행).

**최대 리스크.** 콜드스타트가 아니라 **폭의 유혹.** 6개 버티컬을 동시에 얕게 채우면 비교 깊이가 없어 검색·재방문 모두 실패. → 비컨헤드 1개로 깊이를 먼저 증명.

## 2. 배경 & 문제 정의

### 2.1 왜 지금인가
- **플랫폼 과잉:** 판매채널·프리랜서마켓·크라우드펀딩·공동구매·수출입·배달 등 "붙을 수 있는" 채널 폭증 → 선택 피로 증가.
- **정보 비대칭 고착:** 각 플랫폼은 유리한 정보만 노출(수수료 하한만, 정산주기 애매, 페널티·정산유보 미고지). 중립적 3자 비교의 부재가 구조적으로 지속.
- **AI 검색 전환기:** 생성형 검색이 파편 콘텐츠를 요약 → "구조화된 1차 데이터"를 가진 쪽이 인용·유입 흡수. 지금 스키마를 선점하지 않으면 남의 요약 소스로 전락.

### 2.2 누구의 어떤 페인인가

| 주체 | Job(페인) | 현재 대체행동 | 공백 |
|---|---|---|---|
| **이용 사업자(수요, 1차)** | "내 상황에 맞는 판매/조달/자금 채널을 최소 리스크로 고르고 싶다" | 블로그·유튜브·카페·지인 | 비교축 제각각, 수수료·정산 실체 불투명, 최신성 낮음 |
| **개인(수요)** | "프리랜서마켓/펀딩 어디에 올릴지 고르고 싶다" | 커뮤니티 검색 | 광고성 후기, 정산·페널티 정보 부재 |
| **플랫폼 운영자(공급, 2차)** | "우리 채널에 맞는 셀러/제휴처를 저비용으로 확보하고 싶다" | 직접 영업, 광고 | 발견 채널 부재, BD 인건비 과다 |

> **정량화의 정직한 한계.** 위 페인의 국내 공개 통계는 빈약하다. 따라서 초기엔 **검색 수요(키워드 검색량)**를 관측 가능한 프록시로 사용한다(예: "쿠팡 수수료", "스마트스토어 정산주기", "와디즈 수수료" 롱테일 합). 실제 수치는 런칭 전 키워드 리서치로 확정 — 지금 숫자를 지어내지 않는다.

## 3. 시장 분석 & 경쟁 지형 (3개 레이어)

### 3.1 레이어별 수요·대체재·공백

| 레이어 | 수요 실재성 | 주요 대체재 | 공백(우리 자리) | 화폐화 난이도 |
|---|---|---|---|---|
| **1. 발견·비교** | 높음(검색 관측) | 블로그/유튜브 · G2·Capterra(내부SW만) · 커머스 한정 디렉토리 | 외부 부착형 플랫폼 **범용 비교 스키마 부재** | 낮음(광고/입점) — **먼저** |
| **2. 제휴 매칭** | 중간(운영자 페인 크나 발견 채널 없음) | 직접영업·제휴대행·DM | 상호 송출·정산 **표준 레일 부재** | 중간(정산 수수료) |
| **3. M&A 거래소** | 낮음·희소(고가치·저빈도) | M&A 부티크·사모·개인 브로커·매물 카페 | 소형 플랫폼 특화 거래소 부재 | 높음(거래 수수료·법무) |

### 3.2 경쟁 포지셔닝 맵

```
                내부에서 쓰는 SW ───────────── 외부에 붙는 플랫폼
                (CRM/ERP/회계)                (판매채널/조달/자금)
   범용·구조화   ┌──────────────┐            ┌──────────────┐
   (비교 스키마) │  G2/Capterra │            │   ★ 우리     │  ← 빈 사분면
                └──────────────┘            └──────────────┘
   파편·비구조   ┌──────────────┐            ┌──────────────┐
   (블로그성)    │  IT 리뷰블로그│            │ 커머스채널 후기│
                └──────────────┘            └──────────────┘
```
- **핵심 통찰:** 우측 상단(외부 부착형 × 구조화)이 비어 있다. 단 "넓게 얕게" 채우면 즉시 우측 하단(블로그성)으로 추락. **구조화(깊이)가 곧 진입장벽.**

### 3.3 "폭넓은 버티컬" 결정의 냉정한 평가
- **장점:** SEO 표면적이 넓어 롱테일 유입 총량이 크고, 이음 프로그래매틱 SEO 자산과 궁합 좋음.
- **치명 위험:** 버티컬마다 비교축이 다르다(프리랜서마켓의 "수수료·노출알고리즘" ≠ 수출입의 "관세·통관·에스크로" ≠ 공동구매의 "MOQ·정산유보"). **6개를 동시에 스키마화할 인력 없음**(6~8명).
- **결론:** 폭은 **카탈로그(얕은 목록·기본정보)** 레벨로만 즉시 수용, **비교 깊이**는 **비컨헤드 1개**에 먼저 투입.

## 5. 핵심 가치 제안 & 차별화(해자)

### 5.1 가치 제안
- **수요측:** "흩어진 플랫폼 정보를 **같은 자로 재서** 3분 만에 비교"(특히 숨은 비용·정산리스크 표준화).
- **공급측:** "우리 플랫폼을 정확·유리하게 노출하고, 나중엔 맞는 파트너까지 붙여준다."

### 5.2 방어 가능한 해자 (진입 순서로 두꺼워짐)

| 해자 | 원천 | 모방 난이도 |
|---|---|---|
| 비교 스키마 IP | 버티컬별 정규화 속성·정산리스크 사전 | 높음(도메인 노가다·검증) |
| 다국어 프로그래매틱 SEO | 이음 30개국어·SEO 인프라 재활용 | 높음(인프라·시간) *(주: 1단계 크라우드펀딩(국내중심)엔 기여 없음 — 2단계 매칭·수출입 비컨헤드부터 발현)* |
| 1차 데이터 신뢰 | 운영자 클레임 + 사용자 제보 이중검증 | 중간(운영 리소스) |
| 양면 유동성(1→2 전환) | 수요 트래픽이 공급 광고·제휴 유인 | 높음(선점·네트워크) |

### 5.3 이음 자산 레버리지
- **SEO 인프라:** 버티컬×플랫폼×속성 조합으로 프로그래매틱 페이지 대량 생성 → 롱테일 흡수.
- **다국어:** 수출입·크라우드펀딩 등 국경 넘는 버티컬에서 30개국어가 즉시 차별점.
- **지역×산업 회원 데이터:** 1단계 기여는 적음(솔직히). 단 2단계 "같은 단지/산업 내 매칭"으로 재활성화 가능.

## 6. 서비스 전략 & 단계별 로드맵

### 6.1 확장 논리
```
[1단계 발견·비교] ──트래픽/데이터──▶ [2단계 제휴 매칭] ──신뢰/정산레일──▶ [3단계 M&A 거래소]
```
확장은 자동이 아니다. 명시적 **게이트**를 둔다.

| 전환 | 게이트(가정치, 후 보정) | 미달 시 |
|---|---|---|
| 1→2 | 비컨헤드 MAU ≥ 3만, **슬롯/프로필 아웃바운드 CTR·리드 단가·계산기→외부이동 전환율** 검증, "제휴 문의" 인바운드 관측 | 1단계 깊이·광고 지속 투자 |

> **[v2 정정]** 화폐화 게이트를 '유료 입점 ≥30곳'에서 위 성과지표로 교체함. 국내 크라우드펀딩 모수는 6~20개라 '입점 30곳'은 산술적으로 검증 불가능한 게이트였다. 또한 **'깊이 증명용 비컨헤드(크라우드펀딩)'와 '화폐화용 첫 버티컬(플랫폼 수 많은 유통·프리랜서)'을 분리**한다 — 하나의 버티컬이 깊이와 화폐화를 동시에 달성한다는 암묵 가정을 폐기.
| 2→3 | 제휴 정산 GMV 누적 검증, 매물 인바운드 반복, 법무 자문 확보 | 2단계 확장/추가 버티컬 |

### 6.2 "넓은 카탈로그 + 좁은 비컨헤드" 이중구조 (핵심 베팅)
```
카탈로그(넓게):  프리랜서·크라우드펀딩·배달·수출입·공동구매·유통  ← 얕은 기본정보 + SEO
                       │
                       ▼ (그중 1개만 깊게)
비교(깊게):      ★ 비컨헤드 1개 ★  ← 정규화 속성·정산리스크·검증데이터·에디토리얼·계산기
```
깊이가 증명되면 그 템플릿을 다음 버티컬로 복제.

## 7. 플랫폼 분류 체계(Taxonomy) & 정보구조

### 7.1 분류 축 (다차원 태깅)
1. **부착 목적:** 판매채널 / 자금조달 / 인력·용역 / 조달·구매 / 물류·풀필먼트 / 마케팅·노출
2. **버티컬:** 프리랜서마켓 · 크라우드펀딩 · 배달 · 수출입 · 공동구매 · 유통 …
3. **정산 모델:** 수수료율 / 정산주기 / 정산유보·에스크로 / 광고강요 여부
4. **입점 조건:** 사업자유형 / 카테고리 제한 / 심사기간 / 초기비용
5. **규모·타깃:** 국내/해외, B2B/B2C, 거래규모대
6. **지역(이음 자산):** 단지·권역 태그(2단계 매칭용)

### 7.2 비교 속성 & 데이터 스키마 (여기서 승부)

**공통 스키마:**
```json
{
  "platform_id": "string",
  "name": "string",
  "vertical": ["string"],
  "purpose": ["string"],
  "region": ["string"],
  "languages": ["string"],
  "fee_model": { "type": "percent|flat|mixed", "rate_min": 0, "rate_max": 0, "hidden_fees": [] },
  "settlement": { "cycle_days": 0, "holdback_pct": 0, "escrow": false, "dispute_policy": "string" },
  "onboarding": { "biz_type_required": "string", "categories_allowed": [], "review_days": 0, "setup_cost": 0 },
  "scale": { "b2b_b2c": "string", "geo": "string", "traffic_tier": "관측|추정(근거표기)" },
  "evidence": { "source": "운영자|공식약관|사용자제보", "verified_at": "date", "confidence": "high|mid|low" }
}
```
**버티컬 확장 필드(예):**
- 크라우드펀딩: `success_fee_pct`, `pg_fee`, `refund_policy`, `funding_type(리워드/투자)`
- 수출입: `incoterms_지원`, `통관대행`, `결제보증(TT/LC/에스크로)`, `대상국`
- 프리랜서마켓: `노출알고리즘`, `등급제`, `구매자수수료 별도`

> **데이터 무결성 원칙:** 모든 수치에 `evidence.source`·`verified_at` 강제. "근거 없는 수수료 표기"는 스키마 레벨에서 금지 → 이것이 블로그성 콘텐츠와의 신뢰 격차.

### 7.3 비컨헤드 선정 기준
`(검색 수요) × (정산리스크 정보 비대칭=유틸리티) × (이음 자산 적합도) ÷ (데이터 획득 난이도)`
- 유력 후보(런칭 전 검증 확정): **크라우드펀딩**(수수료·성공수수료·PG·환불 복잡, 플랫폼 수 유한→획득 쉬움) 또는 **수출입**(다국어 자산 강함, 정보 비대칭 극심, 단 스키마 복잡).

## 8. MVP 범위 & 핵심 기능 정의 (1단계)

### 8.1 넣는 것 / 빼는 것

| 구분 | 넣는다(Must) | 뺀다(Not now) |
|---|---|---|
| 콘텐츠 | 비컨헤드 1개 깊은 비교 + 나머지 5개 얕은 카탈로그 | 6개 전부 깊은 비교 |
| 비교 | 2~4개 나란히 비교 테이블 + 정산리스크 하이라이트 | 개인화 추천엔진/AI 상담 |
| 탐색 | 버티컬·목적·조건 필터, 프로그래매틱 SEO 랜딩 | 로그인 필수 기능 |
| 신뢰 | evidence 표기, 오류 제보 버튼 | 풀 UGC 리뷰(어뷰징 부담) |
| 유틸리티 | 수수료·실수령 계산기(정산유보 반영) | 계정연동/실거래 트래킹 |
| 화폐화 | 입점/상위노출 광고 슬롯(수동 판매) 자리 확보 | 셀프서브 광고입찰 |
| 공급측 | 운영자 정보정정 클레임 폼(무료) | 운영자 대시보드 |

### 8.2 핵심 화면
1. **버티컬 허브(프로그래매틱):** `/{vertical}/플랫폼-비교` — SEO 랜딩, 목록·필터.
2. **비교 테이블:** 2~4개 선택 → 같은 축, 숨은비용·정산리스크 배지.
3. **플랫폼 상세:** 스키마 전 항목 + evidence + 제보 버튼 + (광고주) 강조 슬롯.
4. **계산기(단면 유틸):** 매출/모금액 입력 → 수수료·정산유보·실수령·정산일. **로그인 불필요.**
5. **운영자 정정 폼:** 데이터 오류 신고(공급측 첫 접점).

### 8.3 단면(single-player) 유틸리티 = 콜드스타트 완화
- 계산기·비교 테이블은 **다른 사용자 없이도 즉시 유용** → 네트워크 없이 첫 가치 제공.

## 9. 콜드스타트 전략 (정면 대응)

### 9.1 문제 정의(정직하게)
- 1단계의 진짜 콜드스타트는 "광고주"가 아니라 **콘텐츠(데이터) 공급**이다. 데이터는 **우리가 직접 시딩** 가능하므로 닭-달걀이 아니라 **노동 문제** — 이게 이 서비스의 행운.

### 9.2 공급(데이터) 확보 — "우리가 만든다"
1. **에디토리얼 시딩:** 비컨헤드 1개를 손으로 완전 스키마화(플랫폼 15~40개), 공개약관·요금표·운영자 확인.
2. **프로그래매틱 확장:** 나머지 5개는 공개정보 기반 얕은 카탈로그를 SEO 인프라로 대량 생성.
3. **운영자 정정 루프:** 정정 폼으로 운영자가 스스로 품질 향상(무료·상호이익).
4. **다국어 자산:** 해외 플랫폼도 30개국어 커버 → 데이터 폭 저비용 확장.

### 9.3 수요(트래픽) 확보 — SEO 우선
1. **프로그래매틱 SEO 주엔진:** "○○ 수수료/정산주기/vs" 롱테일을 버티컬×플랫폼×속성 조합으로 대량 랜딩.
2. **비교·계산기의 검색 친화성:** "실수령 계산" 류는 검색·공유·백링크가 잘 붙음.
3. **커뮤니티 시딩:** 셀러·창작자 카페/오픈채팅에 계산기·비교표를 답변형으로 배포(스팸 아닌 유틸 제공).
4. **AI 검색 대비:** 구조화 데이터(스키마 마크업)로 생성형 검색 인용 소스화.

### 9.4 왜 "폭"이 콜드스타트를 위협하는가 — 완화책
- 위험: 6개 얕은 버티컬 = 검색 진입해도 "블로그보다 나을 게 없음" → 이탈·재방문 실패 → SEO 신호 악화.
- 완화: 비컨헤드에서 "우리만의 깊이"를 먼저 증명(체류·비교완료·재방문↑) → 검증 후 템플릿 복제. 얕은 5개는 "존재하되 기대치 관리".

### 9.5 광고 화폐화의 콜드스타트
- 광고는 트래픽 선행 필요 → MVP엔 슬롯 자리만, 비컨헤드 MAU 게이트(수만) 넘으면 수동 세일즈로 첫 광고주 확보. 셀프서브는 나중.

### 10.2 개략 유닛 이코노믹스 (가정 명시 — 실측 대체 골격)

- **2단계 제휴(가정) [v2 정정]:** ~~GMV take-rate 3~10%~~ → **제휴 성사 건당 정액 중개료(예: 건당 50~500만원) + 매칭 구독료**로 재정의. 이유: GMV%를 걷으면 정산금 흐름에 개입한 것으로 간주돼 전자금융거래법 등록의무·특금법 리스크가 생기고, §12의 '직접 정산 회피' 원칙과 자기모순. 정액 중개료는 (1) 자금 미접촉으로 규제 회피, (2) 흑자에 필요한 최소 성사 건수를 게이트에 수치로 박아 검증 가능하게 함. **'우리는 결제망·정산금·에스크로에 일절 접근하지 않는다'**는 자금 미보유 원칙을 아키텍처·계약으로 문서화.
- **[v2 정정] 손익분기 분모:** '유료 입점 60~150곳'의 분모를 **비컨헤드 광고주가 아니라 '전 버티컬 카탈로그 광고주 풀'**로 재정의. 크라우드펀딩 단독으로는 손익분기의 10~15%도 못 채움을 적시. ARPU는 '월 전환 셀러 수 × 지불용의액'으로 bottom-up 역산하고, 화폐화 CAC(수동 세일즈 BD 인건비÷성사)를 수요 CAC와 별도 항으로 관리 — 저단가 롱테일은 **셀프서브 자동화가 유닛 성립의 전제조건**.
- **CAC:** SEO 주도라 유료 CAC 낮게 설계(콘텐츠·인프라 고정비로 치환). **핵심 비용은 마케팅이 아니라 데이터 큐레이션 인건비.**

> **냉정한 결론:** 광고는 "첫 현금"이지 "사업의 엔진"이 아니다. 광고만으로는 팀을 못 먹인다. 광고는 2단계 제휴 매출로 가는 다리로 취급하고, KPI도 광고매출보다 "트래픽·데이터 신뢰·제휴 인바운드"를 앞에 둔다.

## 11. 성공 지표 (North Star + KPI)

**North Star:** 주간 *"의사결정 도달 비교 세션"*(≥2개 플랫폼 비교 + 상세/외부이동/계산 실행).

| 경쟁 | 대형 미디어/포털 진입 | — | 스키마 IP·다국어·양면 유동성 선점, 단순 콘텐츠는 방어 불가 인정 |

## 부록 B. 에이전트팀 토론 결과 & v2 개선 (2026-07-01)

> 5개 전문 렌즈(GTM·수익모델·제품·SEO·법무) 병렬 비평 → 이견 토론·종합. 총 36개 지적에서 도출.

### B-1. 렌즈 간 이견과 판정(토론)

**1. MVP 재정의: '데모 UI'인가 'SEO 가설 검증기'인가 — SSG/프리렌더에 지금 투자할 것인가**
- 대립: SEO·데이터·기술 렌즈와 GTM 렌즈는 정면으로 'MVP는 100% 클라이언트 JS 렌더라 색인 불가, pSEO(유일 수요엔진)를 0으로 검증불가 → 지금 Next/Astro/11ty SSG로 전환'을 요구(effort high). 반면 제품·UX 렌즈의 다수 지적(검색창·재방문훅·비교담기 넛지·계산기 정확도)은 현 정적 아키텍처 위에서 즉시 가능한 저비용 개선이라, 아키텍처 대공사가 이 개선들을 지연시킬 수 있음.
- **판정:** 둘 다 옳지만 순서가 핵심. 판정: SSG 풀마이그레이션(high effort)은 지금 하지 말고, '빌드 스크립트 프리렌더'라는 중간 해법을 택한다. platforms.js를 단일 소스로 두고 Node 빌드 스크립트로 (a) /crowdfunding/{id}.html 플랫폼 정적 페이지, (b) /compare/{a}-vs-{c}.html 15개 pair 정적 페이지, (c) sitemap.xml/robots.txt/JSON-LD를 생성한다. 이는 프레임워크 도입 없이 순수 정적 사이트 성격을 유지하면서 '색인 가능한 고유 URL + 구조화 데이터'라는 load-bearing 가설을 실측 가능하게 만든다. UX 개선은 이 프리렌더 위에 하이드레이션으로 병행. 결론: SEO 렌즈의 '지금 해야 한다'가 이기되, 수단은 프레임워크가 아니라 빌드 스크립트(effort를 high→med로 낮춤).

**2. 비컨헤드 확정: 크라우드펀딩(확정 실행됨) vs 수출입(공식·자산 적합도 우위) vs '2주 검증 스프린트로 재개방'**
- 대립: GTM 렌즈: §7.3 선정공식의 '이음 자산 적합도(30개국어)' 항이 수출입에 유리한데 크라우드펀딩 시드는 킥스타터 빼고 전부 languages:['ko']라 다국어 자산이 유휴 → 공식이 장식이거나 결정이 자산과 배치. 게다가 §7.3은 '유력후보(미확정)'인데 MVP·계산기는 이미 크라우드펀딩 전용 하드코딩. 반대 관점(암묵): 크라우드펀딩은 '플랫폼 수 유한→획득 쉬움'이라 깊이를 12개월 안에 증명하기 가장 쉬운 버티컬이고, 수출입은 스키마(인코텀즈·통관·결제보증)가 이질적이라 6~8명이 감당 난이도 높음.
- **판정:** 판정: 크라우드펀딩을 '깊이 증명용 비컨헤드'로 유지하되, GTM 렌즈의 두 지적을 수용해 두 가지를 강제한다. (1) 기획서에서 '다국어 자산은 1단계 크라우드펀딩(국내 중심)에 기여 없음'을 명시하고 해자 목록의 다국어를 1단계 근거에서 제외(§5.2·§5.3 정정). (2) '깊이 증명용 비컨헤드(크라우드펀딩)'와 '화폐화용 첫 버티컬(플랫폼 수 많은 유통·프리랜서)'을 로드맵에서 분리 — 하나의 버티컬이 깊이와 화폐화를 동시에 하리라는 가정을 폐기. 전면 재검증 스프린트는 매몰비용(이미 시드 완료) 대비 실익이 낮아 기각하되, 다국어를 쓰려면 수출입을 '2번째 비컨헤드'로 우선 배치(현재 §13 Q3의 2nd 비컨헤드 슬롯).

**3. 손익분기 분모: 비컨헤드(크라우드펀딩) 광고주 vs 전 버티컬 카탈로그 광고주 vs ARPU 재설정**
- 대립: 수익모델 렌즈: §10.2 '유료 입점 60~150곳'은 국내 크라우드펀딩 전수 10~20개와 산술 모순(모수의 3~10배). GTM 렌즈: §6.1 게이트 '유료 입점 ≥30곳'조차 6개 모수에서 불가능. 두 렌즈가 같은 모순을 지적하나 처방이 갈림 — 수익모델 렌즈는 'ARPU를 40만→150~300만으로 올려 필요 슬롯 6~20곳으로' 또는 '분모를 전 버티컬 카탈로그로', GTM 렌즈는 '게이트를 유료입점 수가 아니라 슬롯 CTR·리드단가·전환으로 교체'.
- **판정:** 판정: 세 처방을 층위로 결합. (1) 화폐화 게이트(§6.1)는 GTM 렌즈대로 '유료 입점 수'에서 '슬롯 CTR·아웃바운드 클릭·리드 단가'로 교체 — 모수 6개 버티컬에서 '입점 30곳'은 검증 불가능한 게이트이므로 폐기. (2) 손익분기 계산은 수익모델 렌즈대로 분모를 '전 버티컬 카탈로그 광고주 풀'로 명시 재정의하고, 크라우드펀딩 단독으로는 손익분기의 10~15%도 못 채움을 기획서에 적시. (3) ARPU는 리드 단가에서 bottom-up 역산(다음 debate 참조). 세 렌즈가 사실상 같은 결론의 다른 면이라 통합에 충돌 없음 — 핵심은 '비컨헤드=화폐화'라는 암묵 가정을 명시적으로 깨는 것.

**4. 화폐화 우선순위: 월정액 슬롯 광고(현 1순위) vs CPL(현 1b) vs 데이터/셀러구독 — '광고 먼저'라는 확정 결정과의 충돌**
- 대립: 수익모델 렌즈: (a) ARPU가 미검증이라 CPL(리드당 과금)을 1순위로 승격하면 볼륨 미검증 상태에서도 매출-가치 연동이 자동 성립. (b) 모든 수익원이 공급측(플랫폼 운영자) 지불에 의존하는데 §0은 '수요측 우선'이라 타깃-수익 전략이 어긋남 → 셀러 프리미엄 구독 등 수요측 직접 과금 라인 필요. (c) 수동 세일즈 전제 하 슬롯 영업 CAC(BD 인건비÷성사)가 슬롯 매출을 잠식 → 저단가는 셀프서브 필수. 확정 결정 관점: 사용자가 '광고·입점/상위노출'을 수익화 1순위로 확정함.
- **판정:** 판정: '광고 먼저'라는 확정 결정은 존중하되 '광고=월정액 슬롯'이라는 좁은 해석을 CPL 우선으로 재정의한다. 근거: MVP 단계엔 트래픽·전환 볼륨이 미검증이라 월정액 슬롯은 지불의사 리스크가 크고, CPL은 아웃바운드 클릭 계측만 붙이면 '가치가 증명된 만큼만 과금'이라 콜드스타트에 강함. 따라서 §10.1을 '1: CPL/아웃바운드(성과형) → 1b: 프리미엄 프로필/상위노출 슬롯'로 순서 교체. 수요측 직접 과금(셀러 구독)은 §0 '수요측 우선'과의 정합성 때문에 반드시 후보에 추가하되, MVP에선 '지불의사 신호 수집(관심 버튼)'까지만. 셀프서브 광고는 유닛 성립의 전제조건이므로 로드맵상 'Q4 이후'에서 '화폐화 흑자 전제'로 격상 표기.

**5. 2단계 수익 표현: GMV take-rate 3~10%(엔진) vs 규제회피(리포팅 수수료) — 자기모순**
- 대립: 수익모델 렌즈: 고정비 6억/년을 take-rate로 커버하려면 제휴 GMV 연 60~200억 필요. 그러나 §12는 규제회피 위해 '직접 정산 회피, 리포팅 수수료만'이라 명시 → 실효 요율 0.5~1%로 떨어져 필요 GMV 600~1200억으로 폭증, 12개월·6~8명과 전혀 안 맞음. 법무 렌즈: GMV%를 걷는 순간 정산 데이터·금액에 개입 인상 → 전자금융거래법 등록의무·특금법 리스크. 둘 다 'GMV take-rate와 규제회피는 양립 불가'로 수렴.
- **판정:** 판정: 두 렌즈가 독립적으로 같은 결론에 도달했으므로 confidence 높음. GMV take-rate 표현을 삭제하고 '제휴 성사 건당 정액 중개료(예 건당 50~500만원) + 매칭 구독료'로 유닛 재정의한다. 이는 (1) 자금 흐름 미접촉으로 전자금융업 등록 리스크 회피, (2) 흑자에 필요한 최소 성사 건수를 §6.1 게이트에 수치로 박아 검증 가능한 가설로 낮춤, 두 목표를 동시 충족. '흑자는 2단계'라는 단정은 '2단계가 흑자를 내려면 연 N건 성사 필요'라는 조건부 가설로 하향. 이 판정은 기획서 §10.1·§10.2·§12를 동시에 수정해야 하는 크로스 섹션 변경.

**6. Q1 실행 순서: 깊이 먼저 검증(현 로드맵) vs SEO 파이프 병행(GTM·SEO 렌즈)**
- 대립: 현 §13: Q1=깊이(비교완료율/재방문), Q2=SEO 색인/롱테일. GTM 렌즈: 유입 자체가 SEO 의존인데 Q1에 SEO 파이프(색인·프로그래매틱 URL)가 없으면 재방문 표본조차 안 모여 3개월간 지표 공전 → 순서가 반대. SEO 렌즈: Q1에 pSEO 색인·CTR 검증 게이트가 아예 없음. 반대 관점: 깊이 없는 SEO 페이지는 thin content라 색인시켜도 강등되므로 깊이가 선행돼야 한다는 현 로드맵 논리도 일리.
- **판정:** 판정: '깊이 vs SEO 순서' 자체가 잘못된 이분법. 둘은 병행돼야 하며, 깊이(콘텐츠)와 SEO(파이프)는 같은 페이지의 앞뒤면이다. Q1을 '깊이 15~40개 + 프로그래매틱 URL·sitemap·JSON-LD 동시 출하'로 재정의하고 Q1 게이트를 2층으로: (1) 4~6주차 '색인 페이지 존재 + 첫 롱테일 노출'(파이프 검증), (2) 12주차 '색인 유입에서 비교완료율/재방문이 블로그 대비 우위'(깊이 검증). 색인 지연(수개월) 동안의 트래픽 공백은 커뮤니티 시딩(§9.3-3)을 Q1으로 앞당겨 브릿지. 단 thin content 우려는 정당하므로 카탈로그 스텁은 Q1에 noindex, 깊은 비컨헤드 페이지만 색인 개방(단계적 색인 정책).

**7. 실명 부정 데이터 공개: '데모 배너로 면책' vs 'confidence!=high면 렌더 차단' — 법적 노출**
- 대립: 법무 렌즈(강경): platforms.js는 와디즈에 '심사 반려 사례 많음', 킥스타터 '목표 미달 시 전액 무산'을 실명·confidence:low로 담고 compare.html L66이 이를 red 'bad' 배지로 자동 공표 → 정보통신망법 §70·형법 §307·업무방해·부정경쟁방지법 소지. 면책 배너는 위법성조각 사유 아님. 처방: confidence!==high면 실명+부정서술 렌더 자체를 차단(현 데이터 전부 low라 배포 시 노출 0이 정상), 데모 단계는 가명 처리. 완화 관점(제품): 이 부정 정보(정산리스크 폭로)야말로 §5.1 '같은 자로 잰다'의 유일한 차별 유틸리티라 전부 숨기면 제품 가치가 사라짐.
- **판정:** 판정: 법무 렌즈가 우선한다 — 공개 URL에 실명+미검증 부정 단정을 두는 것은 감수 불가 리스크이고, '데모라고 밝혔다'는 오히려 '저신뢰 데이터임을 알면서 실명 게시'한 고의 증거가 될 수 있다. 즉시 조치: (1) 렌더 가드 — evidence.confidence!=='high'이거나 source가 공식약관/요금표가 아니면 cautions·부정 배지를 렌더하지 않도록 compare/platform/index에 공통 가드 추가(현 데이터 전부 low → 배포 시 부정 서술 노출 0). (2) '사실(요율 수치, 원문 인용+링크)'과 '평가('반려 많음' 같은 표본 없는 단정)'를 분리 — 평가성은 표본 근거 없이 게시 금지. (3) 제품 가치(정산리스크 비교)는 '평가'가 아니라 '출처 링크 달린 사실 수치'로 전달하면 유지되므로 제품 렌즈의 우려와 양립. 결론: 부정 정보를 없애는 게 아니라 '근거 있는 사실'로만 표현 — 이게 오히려 §7.2 evidence 원칙과 정합.
